Runmark keeps tracked request intent in `runmark/`, Git-ignored runtime state in
`runmark/artifacts/`, and exposes the same execution model through both a CLI
and an MCP server.

## Install

Use the install shape that fits your repo:

### Global install

```bash
npm install -g @exit-zero-labs/runmark
runmark --version
```

### Repo-local / CI install

```bash
npm install --save-dev @exit-zero-labs/runmark
npx runmark --version
```

If you chose the repo-local install above, prefix the commands below with `npx`
(for example, `npx runmark init`). Global installs can use `runmark` directly.

## 1. Scaffold a project

```bash
mkdir demo-api && cd demo-api
runmark init
```

You should see output similar to:

```json
{
  "rootDir": "/path/to/demo-api",
  "createdPaths": [
    "/path/to/demo-api/runmark/artifacts/.gitkeep",
    "/path/to/demo-api/runmark/artifacts/history/.gitkeep",
    "/path/to/demo-api/runmark/artifacts/sessions/.gitkeep",
    "/path/to/demo-api/runmark/config.yaml",
    "/path/to/demo-api/runmark/env/dev.env.yaml",
    "/path/to/demo-api/runmark/requests/ping.request.yaml",
    "/path/to/demo-api/runmark/runs/smoke.run.yaml",
    "/path/to/demo-api/.gitignore"
  ],
  "nextSteps": [
    "In another terminal: runmark demo start",
    "Then run: runmark run --run smoke"
  ]
}
```

`/path/to/demo-api` stands in for your actual absolute project path. The real `init` output also includes the runtime `.gitkeep` placeholders and the root `.gitignore`.

`runmark init` gives you a small but real starting point: one environment, one
request, one run, and schema-aware YAML files. The scaffolded `dev` env already
points at the bundled demo server on `http://127.0.0.1:4318`.

## 2. Start the bundled demo API

In another terminal:

```bash
runmark demo start
```

Leave it running while you work through the quickstart.

## 3. Validate and inspect the tracked files

```bash
runmark validate
runmark describe --run smoke
```

Validation output looks like:

```json
{
  "rootDir": "./demo-api",
  "diagnostics": []
}
```

What you should see:

- `diagnostics` is empty
- `describe --run smoke` shows a single `ping` request step
- no HTTP has been sent yet

## 4. Execute the run

```bash
runmark run --run smoke
```

Execution output looks like:

```json
{
  "session": {
    "sessionId": "run-<timestamp>-<id>",
    "state": "completed",
    "runId": "smoke",
    "artifactManifestPath": "runmark/artifacts/history/<sessionId>/manifest.json"
  },
  "diagnostics": []
}
```

What you should see:

- `session.state` is `completed`
- `diagnostics` is empty
- runtime evidence now exists under `runmark/artifacts/history/<sessionId>/`

## 5. Add local secrets only when the flow needs them

When a request or run references `{{secrets.alias}}`, add that alias under
`runmark/artifacts/secrets.yaml`:

```yaml
devPassword: swordfish
apiToken: sk_test_123
```

Tracked files can also reference `$ENV:NAME`, but secret literals should stay
out of `runmark/`.
